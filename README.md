# МИСИС: Рекомендательная система корректировки расписания

> **Дипломная работа:** Разработка рекомендательной системы для оперативной корректировки расписания и управления аудиторным фондом НИТУ МИСИС

Система поддержки принятия решений для диспетчеров расписания НИТУ МИСИС. Позволяет оперативно перераспределять аудитории при инцидентах (закрытие корпуса, поломка оборудования), бронировать помещения для мероприятий, отменять и восстанавливать занятия, а также анализировать загрузку аудиторного фонда.

**Источником данных служит расписание Института компьютерных наук (ИКН):** учебные группы, занятия и аудитории взяты из расписания ИКН. Соответственно, аудиторный фонд включает только те помещения, в которых проходят пары института.

## Постановка задачи

При возникновении инцидентов (авария в корпусе, ремонт, мероприятие) диспетчеру необходимо переместить десятки занятий в другие аудитории за минимальное время, при этом:

1. **Аудитория должна вмещать группу** — количество мест ≥ число студентов.
2. **Оборудование должно соответствовать** — лекциям нужен проектор, лабораториям — компьютеры.
3. **Перемещение должно быть минимально болезненным** — желательно оставаться в том же корпусе, на том же этаже, не занилять избыточно большие аудитории.
4. **Лекции с несколькими группами** должны быть размещены в одной аудитории (объединённые занятия).

Ручной перебор вариантов при закрытии корпуса (50+ занятий) непрактичен. Система автоматически находит оптимальное перераспределение с помощью Венгерского алгоритма и ранжирует альтернативы по штрафной функции.

## Алгоритмы и методы

### Штрафная функция (`src/scoring.py`)

Для оценки качества замены аудитории используется аддитивная штрафная функция. Каждому кандидату назначается штраф, сумма компонент которого отражает «болезненность» перемещения:

| Компонент штрафа | Вес | Обоснование |
|-------------------|-----|-------------|
| Смена корпуса | +100 | Логистическое неудобство для студентов и преподавателя |
| Разница этажей | +\|Δэтаж\| × 5 | Вертикальное перемещение |
| Избыточная вместимость | +(вместимость − студенты) | Нецелесообразно сажать 15 человек в зал на 200 мест |
| Лишние компьютеры | +10 | Компьютерные аудитории — дефицитный ресурс |
| Лишний проектор | +5 | Проекторы менее критичны |

Формула штрафа:

```
penalty = (building_changed ? 100 : 0)
        + abs(orig_floor - alt_floor) * 5
        + max(0, alt_capacity - students_count)
        + (unneeded_computers ? 10 : 0)
        + (unneeded_projector ? 5 : 0)
```

### Жёсткие ограничения (hard constraints)

Перед вычислением штрафов аудитории проходят через фильтр жёстких ограничений. Аудитория исключается из списка кандидатов, если:

1. **Вместимость недостаточна** — `capacity < students_count`.
2. **Нет нужного проектора** — занятию нужен проектор, а в аудитории его нет.
3. **Нет нужных компьютеров** — занятию нужны компьютеры, а в аудитории их нет.

Все аудитории попадают в матрицу штрафов. Матрица инициализируется значением `BIG_COST = 10⁹` — аудитории, не прошедшие жёсткие ограничения, сохраняют это значение. Допустимые аудитории получают рассчитанный штраф. Венгерский алгоритм гарантированно не выбирает ячейки с `BIG_COST`. Если занятию не нашлось ни одной допустимой аудитории, оно получает диагноз (мало мест, нет оборудования, все заняты).

### Процент совместимости (match_percent)

Для ранжирования альтернатив штраф конвертируется в процент совместимости:

```
match_percent = (1 − (penalty − min_penalty) / (max_penalty − min_penalty)) × 100
```

100% — наилучший вариант, 0% — наихудший из допустимых.

### Венгерский алгоритм (`src/optimization.py`)

Для массового перераспределения занятий применяется **Венгерский алгоритм** (алгоритм Куна—Мункреса), реализованный через `scipy.optimize.linear_sum_assignment`.

**Суть метода:** задача формулируется как задача о назначениях — даны *n* занятий и *m* свободных аудиторий, необходимо найти взаимно-однозначное назначение с минимальной суммарной стоимостью (штрафом).

**Этапы алгоритма в системе:**

1. **Группировка по тайм-слотам** — занятия, идущие в разное время, не конкурируют за аудитории, поэтому оптимизация проводится независимо для каждого слота (день + время + неделя).
2. **Объединённые занятия** — лекции с несколькими группами (один `lesson_id`) сливаются в одну единицу назначения: студенты суммируются, требования к оборудованию объединяются через OR. Это гарантирует, что все группы лекции попадут в одну аудиторию.
3. **Построение матрицы штрафов** — для каждого занятия строится штраф назначения в каждую свободную аудиторию. Невозможные назначения получают штраф `BIG_COST = 10⁹`.
4. **Решение** — `linear_sum_assignment` находит оптимальное назначение за O(n³).
5. **Диагностика** — неназначенным занятиям присваивается причина (мало мест, нет оборудования, все аудитории заняты).

**Преимущество перед жадным алгоритмом:** жадный выбор лучшей аудитории для каждого занятия последовательно может привести к глобально неоптимальному результату (одно занятие забирает аудиторию, более подходящую другому). Венгерский алгоритм гарантирует глобальный оптимум.

### Поиск слотов восстановления (`src/cancellation.py`)

При восстановлении отменённого занятия система ищет лучший тайм-слот среди 10 следующих рабочих дней. Штраф слота восстановления состоит из трёх компонент:

```
total_penalty = room_penalty
              + day_diff × 50
              + time_diff × 5
              + days_from_start × 30
```

| Компонент | Вес | Обоснование |
|-----------|-----|-------------|
| `room_penalty` | — | Штрафная функция аудиторного назначения |
| `day_diff × 50` | 50 за день | Близость к исходному дню недели (циклическая разность) |
| `time_diff × 5` | 5 за слот | Близость к исходному времени |
| `days_from_start × 30` | 30 за день | Чем раньше — тем лучше (не откладывать надолго) |

При поиске проверяются ограничения: группа свободна, преподаватель свободен, аудитория свободна и подходит по параметрам.

```python
BASE_MONDAY = date(2026, 1, 12)  # отсчётная верхняя неделя
weeks_since = (target_date - BASE_MONDAY).days // 7
week_type = "upper" if weeks_since % 2 == 0 else "lower"
```

## Интерфейс

Приложение состоит из семи страниц, доступных через боковую навигацию Streamlit.

### 1. Инциденты

Обработка закрытия аудиторий и массовое перераспределение занятий.

- Выбор аудиторий для закрытия (отдельные или весь корпус)
- Указание периода закрытия (даты начала и окончания)
- Запуск Венгерского алгоритма для перераспределения
- Просмотр результатов: количество переносов, неназначенные, средний процент совместимости
- Подтверждение и сохранение переносов

<!-- Скриншот: ![Инциденты](docs/screenshots/incidents.png) -->

### 2. Бронирование

Бронирование аудиторий для внеучебных мероприятий.

- Ввод данных мероприятия: название, организатор, число участников, требования к оборудованию
- Выбор даты и времени
- Автоматический поиск подходящих аудиторий (сортировка по минимальному избытку мест)
- Проверка конфликтов с расписанием и другими бронями
- При отсутствии свободных аудиторий — предложение онлайн-формата

<!-- Скриншот: ![Бронирование](docs/screenshots/booking.png) -->

### 3. Отмена и восстановление

Полный жизненный цикл отмены и восстановления занятий.

Три вкладки:
- **Отмена** — по преподавателю, по дисциплине или отдельное занятие. Предварительный просмотр, затем подтверждение.
- **Восстановление** — выбор из активных отмен. Одиночное или массовое восстановление с автоподбором слота.
- **Журнал** — история всех отмен со статусами.

При массовом восстановлении система избегает конфликтов между восстанавливаемыми занятиями, подбирая каждому уникальный слот.

<!-- Скриншот: ![Отмена](docs/screenshots/cancellation.png) -->

### 4. Расписание

Визуализация расписания в виде HTML-таблицы (аудитории × тайм-слоты).

Цветовое кодирование:
- 🔵 Синий — обычное занятие (тёмный — лекция)
- 🟢 Зелёный — перенесённое занятие (IN)
- 🔴 Красный с зачёркиванием — перенесённое OUT
- ⚪ Серый с зачёркиванием — отменённое
- 🟠 Оранжевый — восстановленное
- 🟡 Жёлтый — мероприятие

<!-- Скриншот: ![Расписание](docs/screenshots/schedule.png) -->

### 5. Статистика

Аналитика загрузки аудиторного фонда.

- Сводка по фонду: количество аудиторий, корпусов, групп, средняя вместимость, общая загрузка
- Таблица самых загруженных и наименее загруженных аудиторий
- Тепловая карта загрузки по тайм-слотам (красный ≥ 70%, жёлтый ≥ 40%, зелёный < 40%)

<!-- Скриншот: ![Статистика](docs/screenshots/statistics.png) -->

### 6. Управление

CRUD-управление переносами, бронированиями и отменами.

Три вкладки с фильтрацией:
- **Переносы** — просмотр, фильтрация по дате/дисциплине/аудитории, удаление
- **Бронирования** — просмотр, фильтрация, удаление
- **Отмены** — просмотр, фильтрация, удаление (с удалением восстановленных записей расписания)

<!-- Скриншот: ![Управление](docs/screenshots/management.png) -->

### 7. Экспорт

Экспорт расписания в Excel (.xlsx) с цветовым кодированием.

- Выбор группы или преподавателя
- Генерация файла с двумя листами (верхняя / нижняя неделя)
- Цветовая разметка: синий — обычное, зелёный — перенесённое, серый — отменённое, оранжевый — восстановленное

## Запуск

### Предварительные требования

- Python ≥ 3.12
- Доступ к API `schedule.misis.club` (для первичной загрузки данных)

### Полный запуск (рекомендуется)

```bat
run.bat
```

Скрипт автоматически:
1. Создаст виртуальное окружение `.venv` (если отсутствует)
2. Установит зависимости из `requirements.txt`
3. Проверит наличие `data/schedule.db` — предложит загрузить из API при отсутствии
4. Запустит Streamlit на `http://localhost:8502`

```bat
run_clear.bat
```

Запускает Streamlit напрямую (требуется уже существующее venv и БД).

```bat
update_db.bat
```

Удаляет существующую БД и загружает данные заново из API. **Все переносы и бронирования будут потеряны!**

```bash
# Создание базы данных

python pipeline/build_db.py --groups БИВТ-25-1 ...   # конкретные группы

# Запуск приложения
streamlit run src/app.py --server.port 8502
```

## Бенчмарки

Модуль `src/benchmark.py` сравнивает три стратегии назначения аудиторий:

| Стратегия | Описание | Сложность |
|-----------|----------|-----------|
| **Random** | Случайный выбор из допустимых аудиторий | O(n) |
| **Greedy** (best-fit) | Последовательный выбор аудитории с минимальным штрафом | O(n × m) |
| **Hungarian** | Венгерский алгоритм — глобальный оптимум | O(n³) |

### Встроенные сценарии

| # | Сценарий | Описание |
|---|----------|----------|
| 1 | Одиночное перемещение | 1 занятие из корпуса А |
| 2 | Малый пакет | 5 занятий из корпуса А |
| 3 | Один корпус, один день | Корпус А, понедельник, верхняя неделя |
| 4 | Крупный корпус, один день | Корпус Б, понедельник, верхняя неделя |
| 5 | Крупный корпус, оба типа недель | Корпус Б, понедельник, upper + lower |
| 6 | Крупный корпус, полная неделя | Корпус Б, все дни, верхняя неделя |

## Известные ограничения

1. **Оценочное число студентов** — API расписания не предоставляет реальную численность групп, поэтому количество студентов генерируется случайным образом (20–35 для младших курсов, 15–30 для остальных).

2. **Инференс атрибутов аудиторий** — вместимость, наличие проектора и компьютеров определяются не из реальных данных, а выводятся на основе требований проводимых в них занятий (вместимость = максимальный размер группы/потока, проектор = если есть лекции, компьютеры = если есть лаборатории).

3. **Однопользовательский режим** — приложение не предусматривает авторизацию и одновременную работу нескольких диспетчеров. Каждая операция открывает собственное соединение с SQLite.

4. **Поиск свободных аудиторий без конкретной даты** — при поиске без указания `booking_date` система учитывает все возможные переносы IN и бронирования, что может быть избыточно консервативным.

5. **Исключённые корпуса** — аудитории в корпусах «Онлайн», «Каф. ИЯКТ» и «Спортивный комплекс Беляево» исключены из поиска и не участвуют в перераспределении.
